# Testing Framework — 2026-03-20 (revised 2026-03-21)

> **Authorship note:** This framework was generated by Claude agents during ecosystem research (session 2026-03-20). It reflects the agents' reasoning about what to test and how, informed by ecosystem findings and the project's knowledge files. It does not fully represent the author's intent or intuition about the project. The structure and priorities here are a starting point for discussion, not a settled plan. The author should refine, reorder, or discard tests based on what they actually want to learn.

Structure for testing the OpenLight model against real content and against alternative approaches. Agent-driven: agents write to and read from the system. This is not benchmarking — it is live evaluation that produces data points to process.

## Why Test

The model was stress-tested in the abstract (30 agents, 4 domains). The CLI is built and works. But the model has never been tested with real content at meaningful scale, and it has never been compared empirically against alternatives. The ecosystem research (March 2026) identified concrete alternatives worth comparing. The question: what are the model's practical strengths, weaknesses, and boundaries?

## Framing

**Tests produce data to process, not pass/fail verdicts.** OpenLight is a general-purpose substrate — it serves code integration, multi-agent coordination, branching, time travel, browsers, websites, and agents. A test comparing single-agent prose retrieval is one dimension of a multi-dimensional system. A finding that markdown matches OpenLight on that dimension wouldn't invalidate the system's intent — it would map one boundary.

**Agents are flexible.** They adapt to whatever structure and context they're given. Testing an agent with no context about how to use OpenLight tests the out-of-box experience, not the model. Tests should compare systems with **equal context investment** — the OpenLight agent gets culture/guidance, the markdown agent gets equally good instructions. What matters is whether the structure, when properly used, captures knowledge better than alternatives.

**The embedding agent matters as much as the structure.** Results depend heavily on how well the agent embeds knowledge into whatever system it's using. Two agents embedding the same content into OpenLight will produce different structures. This is a feature of agent-driven systems, not a bug — but it means tests must control for or deliberately explore this variable.

## What We're Exploring

Six dimensions worth understanding through testing:

1. **Discovery through structure** — does scope-as-query help agents find knowledge they didn't know to look for? Or does it mainly help with retrieval of known content? Understanding this maps the model's navigational strength.

2. **Instance/relates in practice** — is the binary distinction clear enough with proper context? Where does it naturally express domain structure, and where does it feel forced? This maps the primitive's practical range.

3. **Structural transferability** — knowledge organized by one agent, navigated by another. Does the dimensional structure serve as shared vocabulary, or does each agent's organization need its own context to navigate? This maps the model's role in multi-agent coordination.

4. **Knowledge evolution** — without weights, how does the system handle changing relevance over time? Does binary membership lead to dimension proliferation, or do commits + remove/re-add handle evolution cleanly? This maps the model's temporal behavior.

5. **Structure vs prose at scale** — at what point does dimensional navigation outperform reading well-organized text? This maps the model's efficiency crossover. A high crossover point doesn't invalidate the model (it serves many purposes beyond prose retrieval) but does inform where structural navigation adds value.

6. **Cross-domain range** — do the same primitives feel natural across different domains, or do some domains strain the model? This maps the model's generality — not to prove it's universal, but to understand where it's strongest.

## Test Structure

Each test lives in its own file. Naming: `test-NNN-short-name.md`. Structure within each test:

```
# Test NNN: Short Name

## Claim Being Tested
What specific claim does this test evaluate?

## Setup
- Content to be modeled (or source material)
- Systems involved (OpenLight, alternatives, or OpenLight-only)
- Agent roles (embedder, reader, judge, etc.)

## Procedure
Step-by-step instructions an agent can follow.

## Measurements
What to record — quantitative and qualitative.

## Results
Filled in after the test runs. Raw data, observations, agent transcripts.

## Analysis
What did we learn? Did the result support or challenge the claim?

## Implications
What changes (if any) to the model, the open questions, or the test plan?
```

## Test Plan — Priority Order

### Phase 1: Foundation (test the primitives)

**Test 001: Instance/Relates in Practice**
Question: with proper context, how naturally does instance/relates map across domains?
Setup: 50 chunk-dimension pairs across varied domains, two agents with clear context classify each.
What we learn: where the distinction is natural, where it's ambiguous, whether ambiguity matters for navigation.

**Test 002: Agent Self-Evaluation**
Question: what does an agent experience when navigating via scope?
Setup: 50 chunks across 8-10 dimensions. Agent gets culture context, completes a task, then structured debrief.
What we learn: qualitative signal — what's natural, what's missing, what surprised the agent.

**Test 003: The Inexpressible**
Question: where are the expressiveness boundaries?
Setup: 15 specific modeling challenges (causal chains, negation, temporal sequences, contradiction, etc.) with an agent that has full context.
What we learn: the boundary map — what fits naturally, what requires workarounds, what genuinely can't be expressed. Informs whether conventions can fill gaps or whether new primitives are needed.

### Phase 2: Comparative (test against alternatives)

**Test 004: Sunward Comparative**
Claim: OpenLight navigation produces better cross-cutting answers than alternatives.
Setup: 25 pieces of Sunward non-profit content modeled in 4 systems. Fresh agents answer 20 questions.
What we learn: which model helps agents answer cross-cutting, discovery, temporal, and operational questions.
Content: already written in `in-depth/comparative-test-protocols.md`.

**Test 005: Blind A/B**
Claim: agents prefer OpenLight over flat files when they don't know which is which.
Setup: same content in two systems, wrapped to hide identity. Agent uses both, reports preference.
What we learn: unbiased preference signal and the reasoning behind it.

### Phase 3: Scale and Evolution (test at realistic size)

**Test 006: The Whisper Game**
Question: how much knowledge survives the embed → store → retrieve chain at scale?
Setup: 100K tokens of real source material. Agent embeds into OpenLight. A fresh agent (who never saw the source) answers questions using only `ol`. A judge agent that saw the source interrogates the fresh agent on fidelity.
Variants: single-pass vs iterative embedding, single vs parallel agents, guided vs unguided embedding. The embedding variants are themselves a rich test — does iterative refinement improve fidelity? Does parallel embedding by multiple agents produce better structure than serial?
What we learn: three things simultaneously — embedding quality (write), structural preservation (system), retrieval fidelity (read). Also: where information is lost and whether the loss is in the embedding, the structure, or the retrieval.

**Test 007: Dimension Explosion**
Question: what happens to dimensional structure as the system grows?
Setup: 20 sessions, 10-30 chunks each, one domain (software project). Agent decides when to create dimensions.
What we learn: growth rate of dimensions, naming drift, navigability at scale. Whether the model is self-governing or needs external discipline.

**Test 008: Evolution Under Pressure**
Question: how does binary membership handle knowledge that changes?
Setup: 40 baseline chunks, then 10 rounds of corrections, contradictions, and supersessions.
What we learn: whether the commit model handles evolution gracefully or whether dimension proliferation emerges as a workaround for missing weights. Directly tests the binary-membership-and-evolution open question surfaced 2026-03-20.

### Phase 4: Advanced (deeper questions)

**Test 009: Discovery vs Retrieval**
Claim: scope-as-query enables discovery, not just retrieval.
Setup: 100+ chunks, 15-20 dimensions. Plant "needles" — non-obvious connections. A fresh agent navigates using only scope.
What we learn: whether dimensional intersection guides the agent to knowledge it didn't know to look for.

**Test 010: Multi-Perspective**
Claim: the dimensional model naturally separates and connects perspectives from different roles.
Setup: 3 agents with different role prompts, shared OpenLight base, each contributes their perspective.
What we learn: whether agents create dimensional separation naturally, and whether a synthesis agent benefits from the structure.

**Test 011: Cross-Domain Universality**
Claim: the same primitives work across radically different domains.
Setup: 6 agents, 6 domains (recipes, legal, music, health, geopolitics, software). Each models 30 items.
What we learn: where instance/relates is natural, where it's forced, and what can't be expressed.

**Test 012: Branch Exploration**
Claim: branching supports genuine intellectual disagreement, not just data isolation.
Setup: two agents take opposing positions on a shared branch point, build out their perspectives, then a third agent synthesizes using diff.
What we learn: whether `ol diff` between branches illuminates substantive disagreement.

## Principles

- **Tests produce understanding, not verdicts.** Every result is a data point about the model's behavior in a specific context. Process the finding; don't let it dictate a binary conclusion about a multi-dimensional system.
- **Equal context investment.** Every system under test gets proper context for its agents. OpenLight agents get culture. Markdown agents get organizational guidance. The comparison is between well-used systems, not between a guided system and an unguided one.
- **Record everything.** Agent transcripts, command sequences, token counts, qualitative assessments. Tests are not replicable if only the conclusion is recorded.
- **One test, one file.** Results go in the test file, not in a summary. The test file is the record.
- **The embedding agent is a variable.** How well the agent populates the system matters as much as the structure itself. Tests should either control for this (same instructions across systems) or deliberately explore it (varying embedding strategies).
- **Agent unfamiliarity is a confound.** Early tests will underrate OpenLight (agents haven't learned scope navigation) and overrate markdown (deeply trained). Account for this in analysis. Longitudinal tests (same agent, multiple sessions) can separate structure quality from agent familiarity.
