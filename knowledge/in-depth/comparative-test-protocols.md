# Comparative Test Protocols — Knowledge System Models

> **Authorship note:** These protocols were generated by Claude agents during ecosystem research (session 2026-03-20). The content (Sunward scenario, question design, modeling instructions) reflects the agents' best guess at useful test material, not the author's design. The concrete content may be useful as raw material; the framing and priorities should be reviewed.

Designed to test the same real-world knowledge across four modeling approaches and reveal what each captures, loses, and enables.

**Systems Under Test:**
- **A — OpenLight** (dimensional): chunks + kv, named dimensions, binary instance/relates, scope-as-query, commits
- **B — Typed-edge graph**: nodes, typed labeled edges (CAUSED_BY, PART_OF, etc.), graph traversal
- **C — Multi-graph (MAGMA-style)**: four parallel graphs (semantic, temporal, causal, entity) over same data, intent-aware routing
- **D — Flat files + links (MemFS-style)**: markdown files in directories, bidirectional links, git versioning

## Protocol 1: Non-Profit Organization (Sunward)

### 1A. The Modeling Test

**Content — 25 pieces of knowledge:**

1. Maria Chen founded Sunward in 2019. Executive Director. Phone: 555-0101.
2. James Okafor, Program Director. Manages youth and community programs. Phone: 555-0102.
3. Board Chair: Linda Huang. Joined the board in 2021.
4. Sunward's mission: "Visible change within 30 days of every program."
5. Summer Youth Program runs June–August. Capacity: 60 students. Budget: $180K.
6. Community Energy Initiative launched January 2025. Partnership with Clearwater Foundation.
7. The Clearwater Foundation provides $250K annually for energy programs. Contact: David Reeves.
8. Maria decided in March 2025 to merge the tutoring and mentoring programs into one "Academic Bridge" program.
9. James opposed the merge — believed separate programs served different populations.
10. The board approved the merge 4-1, with Linda casting the deciding vote.
11. Alumni-to-staff pipeline: three former participants now work at Sunward.
12. Volunteers are tracked by hours, not headcount. Current: 12,400 hours/year.
13. The energy initiative is behind schedule — permits delayed by 3 months.
14. Sunward's culture values "radical transparency" — all financials published monthly.
15. FY2025 budget: $1.2M. 60% programs, 25% admin, 15% fundraising.
16. The academic bridge merger saved $40K but parent satisfaction dropped 15%.
17. Monthly all-hands meetings. Notes from March 2025: discussed volunteer burnout.
18. Maria has a standing rule: no program may run at less than 70% capacity for two consecutive quarters.
19. James believes the 70% rule will kill the poetry workshop (currently at 55% capacity).
20. The poetry workshop has produced three published student anthologies.
21. Partnership with Lincoln High School provides classroom space in exchange for volunteer tutors.
22. Grant application pending with Riverside Community Trust. Decision expected June 2025.
23. Two staff members (Aisha, Marcus) are alumni who came through the youth program.
24. The volunteer burnout problem was linked to the energy initiative pulling volunteers from other programs.
25. Linda privately told Maria she regrets the merge vote, considering reopening discussion at June board meeting.

**Modeling instructions per system:**

**System A (OpenLight):**
```
Dimensions to emerge: people, programs, board, culture, finance, partnerships,
  energy-initiative, academic-bridge, decisions, meeting-notes, volunteers

Model each knowledge item as one or more chunks.
Assign instance/relates membership.
Use kv pairs for structured data (names, phone, budget, dates, capacity).

Example — item 8:
  chunk: "Maria decided in March 2025 to merge the tutoring and mentoring
         programs into one Academic Bridge program."
  kv: {date: "2025-03", type: "decision"}
  instance: [decisions, academic-bridge]
  relates: [people, programs, culture]
```

**System B (Typed-edge graph):**
```
Create nodes for: each person, each program, each organization, each decision,
  each meeting, each partnership.

Create typed edges:
  Maria -[FOUNDED]-> Sunward
  Maria -[DECIDED]-> academic-bridge-merge
  James -[OPPOSED]-> academic-bridge-merge
  academic-bridge-merge -[APPROVED_BY]-> board
  energy-initiative -[FUNDED_BY]-> Clearwater
  poetry-workshop -[THREATENED_BY]-> 70%-rule
  volunteer-burnout -[CAUSED_BY]-> energy-initiative-overload
  Aisha -[ALUMNI_OF]-> summer-youth-program
  etc.

Properties on nodes for structured data.
```

**System C (Multi-graph / MAGMA-style):**
```
Semantic graph: cluster by topic similarity (programs, finance, culture, people)
Temporal graph: order by date (founding 2019 → board vote → merge → satisfaction drop → burnout)
Causal graph: merge-decision -> satisfaction-drop, energy-overload -> volunteer-burnout,
              70%-rule -> poetry-workshop-threat
Entity graph: people, organizations, programs as nodes with edges between them

Each item appears in 1-4 graphs depending on what aspects it carries.
```

**System D (Flat files + links):**
```
Directory structure:
  people/maria-chen.md, people/james-okafor.md, people/linda-huang.md
  programs/summer-youth.md, programs/academic-bridge.md, programs/energy-initiative.md
  programs/poetry-workshop.md
  decisions/academic-bridge-merge.md
  meetings/2025-03-allhands.md
  partnerships/clearwater.md, partnerships/lincoln-high.md
  finance/fy2025-budget.md
  culture/radical-transparency.md, culture/70-percent-rule.md

Each file contains prose + YAML frontmatter.
Bidirectional links: [[maria-chen]] appears in decisions/academic-bridge-merge.md.
```

**What to measure after modeling:**

| Metric | How to measure |
|--------|---------------|
| **Information preserved** | For each of the 25 items: is every fact recoverable? Rate 0 (lost), 0.5 (partially), 1 (fully). |
| **Structural overhead** | Count: total nodes/chunks/files. Total edges/memberships/links. Ratio of structural elements to knowledge items. |
| **Workarounds needed** | Count items that required a non-natural modeling choice. Document each workaround. |
| **Cross-cutting expressiveness** | Item 24 (volunteer burnout linked to energy initiative pulling from other programs) touches: volunteers, energy-initiative, programs generally, culture. How many structural elements needed to capture this one fact? |
| **Contradiction captured** | Items 8-9 (Maria decided, James opposed). Is the opposition structurally visible or buried in text? |
| **Prescriptive authority** | Item 18 (70% rule). Can the system distinguish this from a description? |

### 1B. The Navigation Test

**Starting point:** You know Maria Chen exists. You need to understand the current state of Sunward.

**Query 1: "What programs does Sunward run?"**
- Expected answer: Summer Youth, Academic Bridge (merged from tutoring+mentoring), Community Energy Initiative, Poetry Workshop.
- Measure: How many steps to get a complete list? Does any system omit the poetry workshop (it's not linked the same way as the others)?

**Query 2: "What's the financial picture?"**
- Expected answer: $1.2M budget, 60/25/15 split, $250K from Clearwater, merge saved $40K, pending Riverside grant.
- Measure: How many steps? Does the merge savings appear (it lives with a decision, not with finance)?

**Query 3: "Who is unhappy and why?"**
- Expected answer: James (opposed merge), parents (satisfaction dropped), Linda (regrets vote), volunteers (burnout), James again (70% rule threatens poetry workshop).
- Measure: This is deeply cross-cutting. How many queries/traversals/scope compositions needed? Does any system surface all five?

**Query 4: "What caused the volunteer burnout?"**
- Expected answer: Energy initiative pulled volunteers from other programs.
- Measure: Is this a single hop or does it require reading multiple items and inferring?

**Query 5: "What might happen at the June board meeting?"**
- Expected answer: Linda may reopen the merge discussion. The Riverside grant decision is also expected. The poetry workshop's fate depends on the 70% rule.
- Measure: This requires connecting three separate forward-looking items. Does the system structure help or does it require reading everything?

**Scoring per query:**
- Steps to answer (fewer is better)
- Irrelevant information encountered (less is better)
- Unexpected useful connections surfaced (more is better)
- Completeness of answer (0-100%)

### 1D. The "Surprise Me" Test

**Setup:** An agent that did NOT build the knowledge base is given access and asked: "Tell me something interesting about how decisions affect people at Sunward."

**What to measure:**
- Does the system's structure guide the agent toward the merge-decision → parent-satisfaction → James-opposition → Linda-regret chain?
- Does the agent discover the 70% rule → poetry workshop → student anthologies chain (a rule threatening something that produces real value)?
- Does the agent stumble into the volunteer burnout → energy initiative causal chain?
- How much of the knowledge base does the agent need to read before producing an interesting finding?
- **Key question for each system:** Does structure create on-ramps for serendipity, or does it silo information into predictable containers?

**Specific structural predictions:**
- System A: Scoping to `decisions` reveals connected dimensions (people, culture, programs). The connectivity map may surface that decisions and culture are tightly linked — prompting exploration.
- System B: Traversing from a "decision" node follows typed edges (OPPOSED, APPROVED_BY) but may not surface the cultural dimensions unless the agent follows multiple hops.
- System C: The causal graph surfaces decision→consequence chains directly. But the connection between the poetry workshop's value and the 70% rule requires bridging causal and entity graphs.
- System D: Files in `decisions/` link to people and programs. But the poetry workshop's value is in a separate file — the agent finds it only if it follows links broadly or searches.

### 1E. The Strain Test

**Causal chain:** Decision to merge → satisfaction dropped → Linda regrets → possible June board reversal. Additionally: energy initiative → volunteer overload → burnout.
- A: Causality is not a primitive. These items share dimensions (decisions, culture, people) but the direction is in the text, not the structure.
- B: Typed edges (CAUSED_BY, LED_TO) capture direction. This is B's native strength.
- C: The causal graph captures this directly. Strongest system for this strain.
- D: Links are bidirectional and untyped. Causality lives in prose only.

**Temporal sequence:** Founding 2019 → programs established → merge decision March 2025 → board vote → satisfaction drop → burnout → June board meeting.
- A: kv pairs with dates. No structural ordering. An agent must read dates and sort.
- B: Edges like PRECEDED_BY or timestamps on nodes. Can be queried but not native.
- C: The temporal graph captures this directly. Second native strength.
- D: Filenames or frontmatter dates. Git log shows when files were created/modified.

**Negation:** "The poetry workshop is NOT part of the Academic Bridge merger" — it's a separate program.
- A: No scope negation. The poetry workshop is `instance` of `programs` but not `instance` of `academic-bridge`. Distinguishable by absence, not by assertion.
- B: No EXCLUDED_FROM edge type (could add one). Absence of an edge conveys nothing.
- C: Entity graph shows they're separate. No explicit negation.
- D: Separate files. No explicit negation.
- **Observation: negation is hard in ALL systems. No system naturally says "X is not Y."**

**Uncertainty:** "We think the Riverside grant will come through but aren't sure." "Linda might reopen the merge discussion."
- A: Lives in chunk text. kv: {confidence: "uncertain"}? Not a structural primitive.
- B: Could add edge type MIGHT_LEAD_TO vs LED_TO. Edge type proliferation risk.
- C: Not natively handled. Uncertain edges in any graph?
- D: Lives in prose naturally. Markdown handles hedging well.
- **Observation: D handles uncertainty best because prose is naturally hedging-capable. Structured systems must either ignore uncertainty or formalize it.**

**Prescriptive authority:** "No program may run at less than 70% capacity for two consecutive quarters."
- A: This chunk can be `instance` of `culture` (it's a rule). But nothing structurally distinguishes it from a description of culture. The poetry workshop's 55% capacity doesn't trigger anything.
- B: Could create a GOVERNS edge from the rule node to all programs. But this is manual and fragile.
- C: Not natively captured. Rules aren't semantic, temporal, causal, or entity.
- D: A file in `culture/`. Nothing enforces it.
- **Observation: prescriptive authority (rules that override) is a gap in all four systems.**

**Hierarchical containment:** Sunward > Programs > Summer Youth, Academic Bridge, Energy Initiative, Poetry Workshop. Then: Academic Bridge < merged from Tutoring + Mentoring.
- A: Programs dimension collects all programs (via `instance`). No containment depth beyond one level without additional dimensions.
- B: PART_OF edges. Composable: Summer Youth -[PART_OF]-> Programs -[PART_OF]-> Sunward. Natural.
- C: Entity graph handles this via hierarchical edges. Natural.
- D: Directory nesting: `programs/summer-youth.md`. Natural for one level. Deeper nesting gets awkward.

## Protocol 2: Software Project

### 2A. The Modeling Test

**Content — 25 pieces of knowledge:**

1. The project is "Tempo" — a real-time collaborative text editor.
2. Architecture: client (React), server (Rust), CRDT engine (separate Rust crate).
3. The CRDT engine uses a tree-based approach (Fugue algorithm).
4. ADR-001: Chose Fugue over Yjs because Fugue handles tree structures natively.
5. ADR-002: Chose WebSocket over WebRTC for signaling because WebRTC P2P complicates presence.
6. Bug #47: Text duplication on rapid reconnect. Severity: high. Status: open.
7. Bug #47 is caused by the reconnect handler not waiting for CRDT sync to complete.
8. Bug #12: Cursor position lost after undo. Severity: medium. Status: closed (fixed in v0.4.2).
9. Feature: Real-time presence (who's editing where). Shipped in v0.3.0.
10. Feature: Offline mode with sync-on-reconnect. Planned for v0.6.0.
11. The offline mode depends on resolving bug #47 first.
12. Team: Alice (lead, full-stack), Bob (CRDT specialist), Carol (frontend), Dave (devops).
13. Alice decided to prioritize offline mode over rich text formatting. Dave disagreed — formatting is what users ask for.
14. Module: `crdt-engine/` — core CRDT logic. ~4,200 lines of Rust. Zero dependencies.
15. Module: `server/ws-handler.rs` — WebSocket connection management. The reconnect bug lives here.
16. Module: `client/hooks/usePresence.ts` — React hook for presence. Tightly coupled to WebSocket transport.
17. Performance: CRDT merge is O(n log n) but with a large constant factor. Documents over 50K chars show latency.
18. The performance issue was discussed in ADR-003 (deferred — not blocking v1.0).
19. Dependency: `tokio` (Rust async runtime). Pinned at 1.36 due to a regression in 1.37.
20. Dependency: `warp` (Rust web framework). Considering migrating to `axum` — warp's maintainer is less active.
21. CI pipeline: GitHub Actions. Build + test + deploy to staging on every push. Production deploys are manual.
22. Test coverage: 89% for crdt-engine, 62% for server, 41% for client. Carol has been writing integration tests.
23. The client test coverage is low because `usePresence.ts` is hard to unit test (WebSocket dependency).
24. v0.5.0 release plan: fix bug #47, refactor reconnect handler, improve test coverage to 75% across the board.
25. Alice suspects the O(n log n) performance ceiling will require a fundamentally different CRDT approach for v2.0.

**Modeling notes — what to watch for:**

- **Causal chains are central.** ADR-001 → chose Fugue → tree-based CRDT → performance ceiling → possible v2.0 rewrite. Bug #47 cause → fix needed for offline mode → blocks v0.6.0.
- **Containment is deep.** Project > modules > files > functions. The reconnect bug is in a specific file in a specific module.
- **Temporal evolution is essential.** Versions (v0.3.0, v0.4.2, v0.5.0, v0.6.0, v2.0) form a timeline. Features and bugs move through states.
- **People-decision-consequence chains.** Alice's prioritization → Dave's disagreement → what gets built when.

**Per-system modeling challenges to document:**

- System A: How to express "bug #47 IS IN server/ws-handler.rs"? This is containment, not dimensional membership. The bug can be `instance` of `bugs` and `relates` to the server module dimension, but "located in" is different from "about."
- System B: Natural for containment (LOCATED_IN) and causality (CAUSED_BY, BLOCKS). But the relationship between ADR-001 and the performance ceiling is a 4-hop chain — does the structure help or does it just create traversal overhead?
- System C: The causal graph captures decision→consequence. The entity graph captures module containment. Temporal graph captures version timeline. This domain may be C's best case.
- System D: Files for ADRs, bugs, modules, people. Links between them. Version timeline in a `releases/` directory? Or dates in frontmatter?

## Protocol 3: Research Literature Review

### 3A. The Modeling Test

**Content — 25 pieces of knowledge from a literature review on "LLM-based code generation":**

1. Chen et al. 2021 — "Evaluating Large Language Models Trained on Code" (Codex paper). Introduced HumanEval benchmark. 28.8% pass@1.
2. Li et al. 2022 — "Competition-Level Code Generation with AlphaCode." Uses massive sampling (1M solutions) + filtering. 34% solve rate on Codeforces.
3. Roziere et al. 2023 — "Code Llama." Open-weights. Fine-tuned from Llama 2. 53.7% on HumanEval.
4. Lozhkov et al. 2024 — "StarCoder2." 3-15B parameters. Trained on The Stack v2 (619 languages). 46% on HumanEval.
5. Method: few-shot prompting with examples (used by Chen et al., standard since GPT-3).
6. Method: fill-in-the-middle (FIM) training — model predicts missing code span. Used by Code Llama, StarCoder2.
7. Method: instruction tuning — fine-tune on (instruction, code) pairs. Used by Code Llama-Instruct.
8. Finding: FIM training does not degrade left-to-right performance (Code Llama paper, Section 4.2).
9. Finding: larger models show emergent planning — they write "setup" code that only makes sense given the later solution (AlphaCode paper, Appendix C).
10. Contradiction: Chen et al. claim HumanEval measures "functional correctness." Austin et al. 2021 argue the problems are too simple — they measure pattern matching, not understanding.
11. Contradiction: Li et al. claim massive sampling is key to performance. Roziere et al. show a single-pass model (Code Llama) nearly matches with 1000x fewer samples.
12. Open question: Does scaling model size improve code quality, or just code likelihood? No controlled study isolating this variable.
13. Open question: How do these models handle specifications that are deliberately ambiguous? No benchmark tests this.
14. Dataset concern: The Stack v2 may contain test cases from HumanEval in its training data. Contamination is suspected but unproven.
15. Author: Mark Chen (OpenAI) — on Codex and GPT-4 technical report.
16. Author: Yujia Li (DeepMind) — led AlphaCode.
17. Author: Baptiste Roziere (Meta) — led Code Llama.
18. HumanEval: 164 Python problems. Pass@k metric. Widely used but criticized (see item 10).
19. MBPP (Austin et al. 2021): 974 Python problems. Simpler than HumanEval. Less used.
20. The field moved from prompting (2021) → fine-tuning (2022-2023) → instruction tuning (2023-2024). This trajectory mirrors the broader LLM field.
21. Practical finding: code generation works best for "plumbing" code (boilerplate, API calls), worst for algorithmic reasoning (competitive programming).
22. Code Llama's 53.7% HumanEval was state-of-the-art for open models in 2023. GPT-4 scored 67% at the same time.
23. StarCoder2 is trained on explicitly permissive licenses only. This is a differentiator for commercial use.
24. Chen et al.'s "Codex" was the basis for GitHub Copilot's original model.
25. There is no standard benchmark for multi-file code generation as of 2024. All benchmarks test single-function or single-class problems.

**Modeling notes — what to watch for:**

- **Citation relationships.** Paper A cites Paper B. Paper C contradicts Paper A's finding. These are typed, directed relationships.
- **Methodological similarity.** FIM training is a method used by multiple papers. Few-shot prompting is another. These are shared attributes, not edges between specific papers.
- **Temporal progression.** The field evolved: 2021 → 2024. Methods, benchmarks, and findings layer.
- **Contradictions are first-class.** Items 10, 11 are explicit disagreements. They must be recoverable as contradictions, not just as separate facts.
- **Open questions are distinct from findings.** Items 12, 13, 25 are gaps in knowledge, not knowledge itself.

**Per-system specific challenges:**

- System A: How does a contradiction live? Two chunks that both `relate` to the same dimension, with text that disagrees? There's no structural signal that they contradict.
- System B: CONTRADICTS edge between nodes. Natural typed edge. But: is "Austin et al. argues HumanEval is too simple" a node? Or an edge property? It's a CLAIM, not an entity.
- System C: Which graph does a contradiction go in? Semantic (disagreement about meaning)? Causal (paper A's claim → paper B's refutation)? It doesn't fit cleanly.
- System D: A file `contradictions.md` or inline notes in each paper's file? Discoverable only if someone thinks to look.

### 3B. The Navigation Test

**Starting point:** You are writing a survey paper on LLM code generation. You have the knowledge base.

**Query 1: "What benchmarks exist and what are their limitations?"**
- Expected: HumanEval (164 problems, criticized as too simple, possible contamination), MBPP (974, simpler, less used), no multi-file benchmark.
- Measure: Are limitations structurally connected to benchmarks, or buried in separate items?

**Query 2: "What do we know about scaling?"**
- Expected: Open question (item 12). Plus evidence: Code Llama (larger, 53.7%) vs StarCoder2 (smaller, 46%). GPT-4 (67%). But no controlled study.
- Measure: Can the system assemble an answer from scattered evidence + an explicit gap?

**Query 3: "Where do researchers disagree?"**
- Expected: Items 10 (HumanEval validity) and 11 (sampling vs single-pass). Plus the open questions (12, 13, 25).
- Measure: Does the system have a structural concept of disagreement? Or must the agent read everything and detect disagreement in text?

**Query 4: "What's the relationship between Code Llama and Copilot?"**
- Expected: Codex (Chen et al.) → Copilot. Code Llama (Roziere et al.) is Meta, not OpenAI. No direct relationship. But same benchmark (HumanEval), competing approaches.
- Measure: Does the system correctly convey NO direct relationship while still showing meaningful comparison?

**Query 5: "What should I study next based on gaps in this review?"**
- Expected: Multi-file benchmarks (gap), ambiguous specifications (gap), scaling controlled study (gap), contamination proof/disproof (gap).
- Measure: Are gaps/open questions structurally distinct from findings? Or does the agent need to infer what's missing?

### 3C. The Evolution Test

**Change 1 — Correction:** Code Llama's HumanEval score was 53.7% at pass@1, but we had omitted that it reaches 79.3% at pass@100. The single number was misleading.

**Change 2 — Contradiction:** A new paper (2024) shows evidence of HumanEval contamination in The Stack v2 — vindicating the suspicion in item 14. StarCoder2's benchmark number is now questionable.

**Change 3 — New connection:** A new paper shows that FIM training (item 6) actually enables a form of planning similar to what AlphaCode achieved with sampling (item 9). Two seemingly different approaches converge on the same capability.

**Change 4 — Removal:** Austin et al. 2021 (MBPP) retracted a subset of their problems due to errors. 74 of 974 problems were flawed. The benchmark still exists but is diminished.

**Change 5 — Split:** "Few-shot prompting" (item 5) needs to be split into "few-shot with examples" and "chain-of-thought prompting" — the review conflated them. They're different methods with different results.

---

### 3D. The "Surprise Me" Test

**Prompt:** "What's the most surprising pattern in this literature?"

**Expected high-quality answers:**
- The massive sampling approach (AlphaCode, 1M solutions) was largely abandoned once single-pass models caught up — suggesting sampling was a brute-force substitute for model capability.
- FIM training doesn't hurt left-to-right performance — suggesting the model's understanding is more general than the training task implies.
- There's no multi-file benchmark despite code generation being used primarily for multi-file projects — the field is optimizing for the wrong thing.

**Measure:** Does the structure enable pattern-finding across papers, or does the agent just retrieve individual facts?

---

### 3E. The Strain Test

**Citation chains:** Chen et al. → Code Llama (builds on Codex approach) → StarCoder2 (alternative open approach). And: Chen et al. → Austin et al. (responds with critique).
- A: Papers share method dimensions. Citation direction is not structural.
- B: CITES, BUILDS_ON, CRITIQUES edges. Natural and directed.
- C: Semantic graph clusters by topic. Temporal graph orders by year. Neither captures citation direction specifically.
- D: Links between files. Citation direction in text only.

**Methodological grouping:** FIM training used by Code Llama AND StarCoder2. Few-shot prompting used by Codex AND AlphaCode.
- A: `methods` dimension with FIM and few-shot as sub-dimensions or chunks. Papers `relate` to the method. Natural grouping.
- B: Nodes for methods, USES edges from papers. Natural but creates intermediate nodes for every method.
- C: Semantic graph clusters papers using similar methods. Implicit, not explicit.
- D: Tags in frontmatter. Requires manual search.

**Contradictions:** Two papers disagree. This is a relationship between claims, not between entities.
- A: Two chunks share dimensions. Contradiction is in text. No structural signal.
- B: CONTRADICTS edge. But between what nodes? The papers? The findings? Need to model findings as separate nodes.
- C: Could be causal graph (claim → counterclaim). Or semantic (disagreement cluster). Awkward in either.
- D: Inline notes or a contradictions file. Natural in prose, invisible in structure.

## Protocol 4: Personal Life Knowledge ("Second Brain")

### 4A. The Modeling Test

**Content — 25 pieces of knowledge:**

1. I'm allergic to shellfish. Discovered age 12. Carry an EpiPen.
2. My partner Alex and I have been together since 2018.
3. Alex is a veterinarian at Eastside Animal Clinic.
4. We're saving for a house. Target: $80K down payment. Currently at $52K.
5. I run 3x/week. Goal: sub-4-hour marathon by October.
6. Current marathon PR: 4:12 (Portland 2024).
7. My knee bothers me after runs longer than 15 miles. Dr. Patel says it's IT band syndrome.
8. Dr. Patel recommended specific stretches and limiting long runs to once per week.
9. Mom's 70th birthday is July 15. Planning a surprise party.
10. Mom lives in Sacramento. Dad passed away in 2020.
11. Sister (Rachel) is helping plan the party. She's flying in from Austin.
12. I should NOT get Mom a Kindle — she mentioned hating e-readers last Thanksgiving.
13. Favorite restaurant: Olea (Mediterranean). Alex and I go for anniversaries.
14. I'm considering switching jobs. Current: senior engineer at Meridian. Unhappy with management.
15. Applied to Volta Labs. Interview on April 3. They use Rust (I want to learn Rust).
16. If I switch jobs, we lose employer health insurance for 60 days. Need to plan COBRA.
17. Alex's clinic is 10 minutes from our current apartment but 45 minutes from the neighborhoods where we want to buy.
18. I promised my friend Sam I'd help him move on March 29.
19. Sam is going through a divorce. Be sensitive when visiting.
20. My book club meets first Thursday of every month. Currently reading "Piranesi" by Susanna Clarke.
21. I owe Rachel $200 from splitting Mom's party venue deposit.
22. Goal: learn conversational Spanish by end of 2025. Using Duolingo (180-day streak).
23. The Volta Labs interview requires a take-home project in Rust. Due April 1.
24. I'm worried the job switch + house purchase timing will be a problem for mortgage approval.
25. Alex doesn't know about the Volta Labs application. I want to discuss it this weekend.

**Modeling notes — what to watch for:**

- **This is the messiest domain.** A shellfish allergy, a marathon goal, a job search, and a mother's birthday party all coexist. The connecting tissue is a single person's life.
- **Temporal urgency.** March 29 (help Sam move), April 1 (Rust project due), April 3 (interview), July 15 (Mom's birthday), October (marathon). Some items have deadlines; others don't.
- **Emotional context.** Sam's divorce, Dad passing, unhappiness at work, Alex not knowing about the application. These aren't tags — they're textures.
- **Cross-cutting concern:** The job switch connects to: health insurance (item 16), mortgage approval (item 24), Rust learning (items 15, 23), Alex's commute (item 17), Alex not knowing (item 25). One decision ripples across multiple life domains.

**Per-system specific predictions:**

- System A: Dimensions might include: health, running, family, finance, career, relationships, goals, people, alex, mom, rachel, sam. The cross-cutting nature of the job switch (career + finance + health + relationships + housing) tests whether dimensional intersection reveals compound concerns.
- System B: Nodes for people, goals, events, decisions. Edge types: RELATED_TO, AFFECTS, SCHEDULED_FOR, OWES. Becomes a dense graph quickly. The emotional context (Sam's divorce) is a property, not an edge — where does it live?
- System C: Semantic graph clusters life domains. Temporal graph captures deadlines. Causal graph: job switch → insurance gap → COBRA planning. Entity graph: people. But "I'm worried" (item 24) doesn't fit in any graph.
- System D: Directories like `people/`, `health/`, `career/`, `finance/`, `goals/`. Cross-cutting items like the job switch need to be in one file with links to many others. Which directory does it go in?

### 4B. The Navigation Test

**Starting point:** It's March 27. You're planning the next two weeks.

**Query 1: "What do I have coming up?"**
- Expected: March 29 (help Sam move), April 1 (Rust project due), April 3 (Volta interview), first Thursday (book club), discuss job with Alex this weekend.
- Measure: Can the system surface time-sensitive items without reading everything?

**Query 2: "What should I know before helping Sam move?"**
- Expected: He's going through a divorce. Be sensitive. Also: March 29 is two days before the Rust project is due — time management concern.
- Measure: Does the system connect the emotional context AND the scheduling conflict?

**Query 3: "How does the job decision affect our house plans?"**
- Expected: 60-day insurance gap (COBRA), mortgage approval timing, Alex's commute gets worse, haven't told Alex yet.
- Measure: This requires traversing career → finance → health → housing → relationship → communication. How naturally does this chain emerge?

**Query 4: "What's my health situation?"**
- Expected: Shellfish allergy (EpiPen), IT band syndrome (Dr. Patel's advice, limit long runs), need to plan COBRA if switching jobs.
- Measure: "Health" spans chronic conditions, sports injuries, and insurance coverage. Does the system unify these or silo them?

**Query 5: "What do I owe people?"**
- Expected: $200 to Rachel, helping Sam move March 29, promised Alex a conversation about the job.
- Measure: "Owe" includes financial debts, physical commitments, and emotional obligations. Can any system capture this breadth?

**Change 2 — Contradiction:** Alex found out about the Volta application (saw an email). Alex is supportive but upset about being kept in the dark. The knowledge about "Alex doesn't know" is now false AND there's a new emotional dynamic.

### 4D. The "Surprise Me" Test

**Prompt:** "What's something I might be overlooking?"

**Expected high-quality answers:**

**Measure:** Does the structure help surface overlooked INTERACTIONS between items that individually seem fine? This is the core test of a "second brain" — not recall, but synthesis across life domains.

### 4E. The Strain Test

**Cross-domain causality:** Job switch → insurance gap → COBRA → financial strain → house savings impact → mortgage approval risk → housing timeline delay → Alex's commute stays bad longer.
- Eight-link causal chain crossing career, health, finance, housing, and relationship domains.
- A: These items share many dimensions. The chain exists but is invisible without reading all items.
- B: AFFECTS/LEADS_TO edges. Expressible but requires modeling all intermediate states.
- C: Causal graph can capture this. But it's interleaved with entity and temporal concerns at every step.
- D: Requires following links through 5+ files. The chain is implicit.

**Emotional/social context:** "Be sensitive when visiting Sam" is not a fact — it's a behavioral directive attached to a person. "Alex doesn't know" is a relational state, not a property.
- A: Lives in chunk text. Maybe kv: {note: "sensitive - divorce"} but that feels wrong.
- B: Could model as SENSITIVE_TOPIC edge. Edge type proliferation.
- C: None of the four graphs are designed for emotional context.
- D: Prose handles this naturally. "Note to self: Sam is going through a lot right now."

**Negation with purpose:** "Do NOT get Mom a Kindle." This is not just negation — it carries a reason (she hated e-readers) and a source (Thanksgiving conversation).
- A: A chunk on gift-ideas dimension with text containing the negation. Not structurally distinguishable from a positive recommendation.
- B: A NOT_RECOMMENDED edge from Mom to Kindle with reason property. Possible but edge type is hyper-specific.
- C: Not capturable in any of the four graph types.
- D: A note in `people/mom.md` under gift ideas. Natural in prose.

## Measurement Framework

### Scoring Rubric

For each test, score each system 1-5:

**1A-4A (Modeling):**
- 5: All knowledge captured naturally with no workarounds
- 4: All knowledge captured, 1-2 minor workarounds
- 3: Most knowledge captured, some required restructuring
- 2: Significant information lost or requiring major workarounds
- 1: The domain fundamentally strains the model

**1B-4B (Navigation):**
- 5: Answers found in 1-2 steps, unexpected connections surfaced
- 4: Answers found in 2-3 steps, no false leads
- 3: Answers found but required 4+ steps or encountered significant irrelevant content
- 2: Answers partially found, important connections missed
- 1: System structure actively hindered finding the answer

**1C-4C (Evolution):**
- 5: Change applied naturally, history useful, no collateral damage
- 4: Change applied with minor restructuring, history preserved
- 3: Change required noticeable restructuring, some history ambiguous
- 2: Change required significant restructuring, history partially lost
- 1: Change broke existing structure or required fundamental redesign

**1D-4D (Surprise Me):**
- 5: Structure actively guided agent to non-obvious insights
- 4: Structure didn't hinder, agent found some interesting connections
- 3: Structure was neutral — agent relied on brute-force reading
- 2: Structure siloed information, making cross-cutting discovery harder
- 1: Structure actively prevented serendipitous discovery

**1E-4E (Strain):**
- Score each strain type (causal, temporal, containment, negation, uncertainty, prescriptive) on the same 1-5 scale per system. This produces a strain profile per system — which strain types each handles well or poorly.

### Expected Outcome Patterns

Based on the test design, the predicted strengths/weaknesses:

| System | Best Domain | Worst Domain | Predicted Strength | Predicted Weakness |
|--------|------------|-------------|-------------------|-------------------|
| A (OpenLight) | Non-profit (cross-cutting concerns) | Software (deep containment, causal chains) | Cross-cutting queries, serendipity, evolution via commits | Causality, containment depth, negation |
| B (Typed-edge) | Software (typed relationships natural) | Personal life (edge type proliferation) | Causal chains, containment, typed traversal | Cross-cutting queries (requires multi-hop), no versioning, edge type explosion |
| C (Multi-graph) | Software (all four graph types exercised) | Personal life (doesn't fit four categories) | Temporal + causal native, structured routing | Emotional context, uncertainty, anything outside the four graph types |
| D (Flat files) | Personal life (prose handles messiness) | Research review (structured relationships lost) | Uncertainty, emotional context, versioning (git), human readability | Structured queries, cross-cutting discovery, anything requiring computation over structure |

### What Constitutes a Meaningful Finding

A test result is meaningful when:

1. **A system fails where it's expected to succeed.** If System B (typed edges) can't capture a causal chain cleanly, that contradicts its theorized strength.
2. **A system succeeds where it's expected to fail.** If System D (flat files) enables better serendipitous discovery than System A, that says something about the value of structural simplicity.
3. **A workaround becomes the norm.** If System A consistently requires "convention" dimensions (a `contradictions` dimension, a `causal-chains` dimension) to handle what other systems do natively, the workarounds are the finding.
4. **Evolution tests reveal hidden coupling.** If a simple change (correction, split) requires touching 5+ structural elements in one system but 1-2 in another, the structural coupling is the finding.
5. **The surprise test differentiates.** If agents exploring the same knowledge produce qualitatively different insights depending on the system, the structure is shaping cognition — for better or worse.

## Execution Instructions

**For each domain (1-4), for each system (A-D):**

1. Model all 25 items. Document decisions, workarounds, and what felt natural vs forced.
2. Run all 5 navigation queries. Record steps, irrelevant hits, unexpected finds, completeness.
3. Apply all 5 evolution changes in sequence. Document restructuring effort and history quality after each.
4. Give the completed, evolved knowledge base to a fresh agent. Run the "Surprise Me" prompt. Record what was found and how.
5. Assess each strain type against the completed knowledge base. Document specific failures and workarounds.

**Total: 4 domains x 4 systems x 5 tests = 80 test executions.**

Each test should take 15-30 minutes when the content is pre-prepared. The modeling test is the longest; the navigation and evolution tests can run from the modeled state.

The systems need not be implemented in software for all tests. The modeling test can be done on paper (what WOULD the data look like in each system). The navigation test requires either a real system or a careful simulation of what queries return. The evolution test reveals the most when applied to a real system. The surprise test requires a real agent.

**Recommended execution order:** Model all four domains in System A first (since the CLI exists), then do the paper-based modeling for B/C/D. Run navigation tests on System A with the real CLI. Simulate navigation for B/C/D. This creates a concrete vs theoretical comparison — which is itself a finding.
